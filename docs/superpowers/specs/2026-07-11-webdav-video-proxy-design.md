# WebDAV 视频网盘代理 — 设计文档

- **日期**: 2026-07-11
- **主题**: webdav-video-proxy
- **状态**: 待用户复审

## 1. 背景与目标

做一个跑在 Docker 容器里的 WebDAV 视频网盘代理，面向本地播放器（HTTP Range 取流），核心解决**播放卡顿、起播慢、拖动卡**。上游是「多源混合」的 WebDAV 服务（典型如 Alist，一个 endpoint 聚合多个真实驱动后端，如阿里云盘/夸克/百度，快慢天差地别）。

代理实现四件事：

1. **资源速率评估** — 对各子源持续测速，产出源画像。
2. **分片提速** — 对支持多连接的子源，单源、单文件、多连接 Range 并行拉取，分片拼回顺序流。
3. **预加载加速** — 顺序预读当前文件 + 预取下一集，提前填缓存。
4. **慢源标记** — 把评估结果标进「影视文件夹名」，让用户在播放器目录导航里看到源速度。

## 2. 关键约束与明确不做

这些约束来自 brainstorming 阶段的反复确认，是设计的硬前提：

- **没有同文件多源副本**：每个文件只存在于一个源上。不存在「同一文件换更快的源取」这步。择优从文件级退化为纯源级。
- **子源 = (endpoint, 顶级路径段)**：一个聚合 WebDAV 按**路径第一段目录**拆成多个子源（如 `/阿里云盘/...`、`/夸克网盘/...`）。驱动内部子路径分不出（路径看不出文件落哪个驱动），但顶层路径段能区分驱动。测速、画像、分片决策、慢源标记全部落到**子源**粒度，不落端点也不落文件。
- **不做影视识别**：同名/同路径但实际是不同码率、不同版本的不同源，代理不去识别「同一部影片的不同版本」，只按文件路径当不同文件处理。没有影视数据库。
- **不真改上游**：绝不调用 `MOVE`/`PROPPATCH` 修改上游真实文件名/目录名。所有「改名」都是代理对外展示的虚拟改名。
- **不写自定义响应头**：本地播放器（含网易爆米花等刮削播放器）不读 `X-*` 响应头，写了无用，故标记不落响应头。
- **不做状态页/查询接口/PROPFIND 自定义属性**：标记的唯一出口是虚拟改文件夹名。
- **不做跨源分片**：分片是「单源、单文件、多连接并行」，不是「不同源各拉一块拼」。用户明确不要跨源。

## 3. 架构总览

方案选择：**A（透传 + 元数据层 + 缓存 + 预加载）为骨架，C（多连接分片并行拉取）作为主路径里的分片提速引擎**，两者融合在同一条请求路径上。B（先缓存后转发的缓冲代理）被否决（起播延迟大，违背核心目标）。

请求主路径（每个 Range 请求依次过这几关）：

1. **路由层** — 该文件在哪个子源（映射固定，无择优）。
2. **缓存层(§2)** — 块级命中检查。全命中直接吐、跳过分片。
3. **分片决策器(§3)** — 首块探速 + 动态扩路，决定开几路。不支持的源降级单连接。
4. **取流器(§3)** — N 路 Range 并行拉分片，乱序到达、顺序吐出。
5. **边收边吐** — 透传给播放器，不整段缓冲再吐。

旁路单元（不挡主路径）：

- **§1 速率评估器** — 子源画像，喂给 §3 决策器和 §5 标记。
- **§4 预加载器** — 顺序预读 + 预取下一集，提前触发 §3 填 §2。
- **§5 标记单元** — 列目录时按影视文件夹名虚拟改名。

```
播放器请求
   │
   ▼
路由层 ──§1 评估器(子源画像)──┐
   │                          │(建议并发度/慢源标记)
   ▼                          │
缓存层(§2)──命中?──是────────►边收边吐──►播放器
   │(否，块未命中)              ▲
   ▼                          │
§3 决策器(首块探速+动态扩路)──┐ │
   ▼                          │ │
§3 取流器(N路Range并行,乱序到顺序吐,回灌缓存)─┘
   │                          │
   └──(吞吐样本)──►§1 评估器──┘
                          ▲
§4 预加载器(顺序预读+预取下一集,节流让步)──调度──►§3取流器
§5 标记(列目录时按影视文件夹名改写)──读§1画像
```

主路径只走 §1→§2→§3→边吐；§4/§5 在旁路，任何失败都透明降级，绝不让播放器断流。

## 4. 单元设计

每个单元边界清晰：做什么、怎么用、依赖什么。

### §1 速率评估器（Rate Assessor）

**做什么：** 对每个子源维护画像，描述带宽档位、多连接友好度、建议并发度、是否慢源。代理唯一对上游做主动探测的单元，其余单元消费其输出。

**子源定义：** 子源 = `(WebDAV endpoint, 顶级路径段)`。

**测速方式（主动 + 被动结合）：**

- **被动测速（主数据源）**：每次真实取流，按文件的顶级路径段归到对应子源，记 `(子源, 吞吐)` 样本到该子源的滚动样本集。零打扰。
- **主动探测（低频，补盲区）**：负责被动摸不到的两件事：
  1. **冷启动画像**：子源首次出现、被动样本为 0 时，主动探测一次给起步画像，避免首次请求瞎猜。
  2. **多连接友好度判定**：被动永远测不出（被动只按既有并发度跑）。对该子源下一个文件分别用 1 路和 N 路拉同样大小块比较。N 路明显大于 1 路 → 友好；N 路 ≈ 1 路 → 不友好（按账号限速）→ 降级单连接；N 路明显更慢或报错 → 风控 → 单连接专用。

主动探测节流：**最小间隔**（同一子源两次主动探测间隔不低于配置值，如 1 小时）；整进程对总上游主动请求量设上限。冷启动单路探速优先，多连接对照可等被动样本积累后再做。

**画像字段（存 SQLite）：**
`子源id, 带宽档位(MB/s), 多连接友好度(友好|不友好|风控|未知), 建议并发度, 慢源标记(布尔), 最近N个吞吐样本, 更新时间戳`

- **慢源定义**：带宽档位低于配置阈值（如 < 2 MB/s），或建议并发度被降为 1 且吞吐仍低。
- **建议并发度**：友好源给配置上限（默认 4），不友好/风控源强制 1。这是 §3 决策器要的输入。

**画像时效：** 最近 N 个样本 + 最大时效双约束（如最近 20 个样本且不超过 6 小时）。样本不足或全过期 → 友好度回退「未知」→ 保守单连接。

**出口：** ①喂 §3 决策器（建议并发度 + 多连接友好度）；②喂 §5 标记（慢源标记 + 带宽档位）。无文件级出口。

**边界：** 只产画像，不路由、不吐数据、不标记。画像失效回退未知，决策器对未知保守单连接。

### §2 缓存层（Cache）

**做什么：** 把从上游取回的字节存到本地磁盘，下次同样 Range 命中就从本地吐、跳过分片取流。A 骨架里唯一持久化字节的单元。

**缓存 key：** `(子源, 真实文件路径, 文件 ETag/Last-Modified)`。必须带版本标识，否则上游文件改了缓存还吐旧数据。

**粒度：块缓存，不整文件缓存。** 每个文件按固定块大小（如 4MB）切，以块为单位存/缺失。块级命中——请求 Range 全落在已缓存块 → 全命中直吐；部分命中 → 命中块从缓存读，缺失块走 §3 回填，回填写回缓存。

**淘汰策略：** LRU 按块淘汰，配磁盘容量上限（可配，Docker 卷）。

**缓存大小管理：**
- **硬上限**（`cache.max-size`，如 50GB）+ **高水位**（90%）/ **低水位**（70%）。达高水位触发主动淘汰，低水位停止。用高低水位避免临界抖动。
- **引用计数保护**：正在被取流/预加载使用的块（引用 >0）进保留集，淘汰跳过。
- **异步淘汰**：达高水位时不阻塞主路径，淘汰丢给后台 worker；主路径只记「需要淘汰」标记即返回。
- **写前检查**：每写一块前检查总量，已到硬上限且后台未及淘汰 → 这块直接不写（跳过缓存，纯透传），宁可不缓存也不超限。
- **冷启动重建**：容器启动扫描缓存目录重建块索引到 SQLite。若磁盘可用空间不足，硬上限自动降到安全比例并日志警告。
- **TTL 兜底**（`cache.ttl`，如 7 天）：单块到期即使没达水位也淘汰，防陈旧块长期占位。

**一致性：** 缓存命中前向上游发轻量 `HEAD`/`PROPFIND` 拿当前 ETag 比对 key 版本；不匹配整文件失效重建。校验低频做（如缓存条目超过 N 分钟才校验一次），不必每次问上游，否则抵消缓存意义。

**与 §4 关系：** 缓存层不主动预读，只提供 `get(range)`/`put(block)` 接口；预加载器调 `put` 提前填块。职责分离——缓存只管存取，预加载器决定填什么。

**边界：** 不评估速度、不路由、不标记，纯字节存取。命中失败/磁盘满 → 透明降级到不分片单路取流，绝不断流。

**配置项：** `cache.max-size`、`cache.block-size`、`cache.high-watermark`、`cache.low-watermark`、`cache.ttl`。

### §3 分片决策器 + 取流器（Segment Fetcher）

方案 C 的引擎。紧耦合整体：决策器决定「怎么拉」，取流器「去拉」。

#### §3.1 分片决策器

**输入：** 子源画像（建议并发度、多连接友好度）、请求 Range、缓存命中情况。

**输出：** 拉取计划 `{起始字节, 结束字节, 并发度 N, 分块大小}`，喂取流器。

**决策逻辑（按优先级）：**
1. 缓存全命中 → 不进决策器，缓存层直吐。
2. 多连接不友好/风控/未知 → `N=1`，单路取流。未知当不友好处理。
3. 多连接友好 → `N = 建议并发度`（默认上限 4，可配）。
4. **首块探速 + 动态扩路（关键）**：因「路径看不出驱动」画像对单次预测力弱。即使画像说友好给 N=4，决策器也先用 N=1 起手拉首块测实际吞吐；实际明显高于画像预期且上游无报错 → 追加到 N；慢/报错 → 保持 1。把「事后才知道速度」转化成「边取边调」。

**分块大小：** 与 §2 缓存块大小对齐（如 4MB），分片正好是一个缓存块，直接 `put` 不二次切分。

#### §3.2 取流器

**输入：** 拉取计划。

**做什么：** 开 N 个 Range 子请求并行拉，每个拉一个 `[block_start, block_end]` 分片。产出**按播放器 Range 顺序排列的 reader**，让边收边吐从头开始吐，不等所有分片到齐。

**关键难点 — 乱序到达、顺序吐出：** N 路并行，先返回的分片未必是播放器要的开头那段。取流器维护**按字节偏移排序的待吐队列**：分片谁先到谁入队，吐出指针从 Range 起始字节往后推，遇到队列里有的块就吐、遇到还没到的块就等那一路；设等待超时，超时降级补一路单连接救该块。这是 C 里最容易写错处，单独测。

**与缓存联动：** 每个分片拉完立刻 `put` 进缓存（块大小对齐直接落块）。下次同 Range 命中缓存跳过整条 §3。

**失败处理：**
- 单路 Range 子请求失败（网络抖动、风控 429）→ 重试该分片，重试次数有上限；重试期间不阻塞已就绪分片吐出。
- 某分片重试耗尽仍失败 → 该分片降级换单路重试，或最终回 502 让播放器重试（已成功块不丢，重试时命中缓存）。
- 上游对该子源连续多请求 429 → 反馈 §1 评估器，把该子源友好度降级为「风控」，后续决策器对它强制 N=1。

**实测回灌：** 整段拉完（或拉到中途评估够稳定）后，把 `(子源, 吞吐)` 样本喂回 §1 更新画像。§1 和 §3 形成闭环。

**边界：** 决策器只产计划不下发数据；取流器只拉不存策略（只 `put` 缓存不决定淘汰）、不标记、不路由。两者合起来只管「按计划拉出顺序流」。

### §4 预加载器（Preloader）

**做什么：** 在播放器还没请求前，提前把「马上要播的内容」拉进缓存，让真正请求到来时命中缓存、跳过分片取流。是 §2 缓存 + §3 取流器的策略驱动者，自己不碰字节。

#### §4.1 顺序预读（当前文件）
- **触发：** 播放器正请求某文件某段 Range（在播），取流器已在拉这段。预加载器盯「当前吐出指针位置」，提前把指针之后、同文件接下来的若干块（如预读 2-3 个块 = 8-12MB）调度给 §3 拉好塞缓存。
- **目的：** 拖动进度条/顺序播放不卡。
- **节流：** 只预读不浪费——播放器跳走（吐出指针跳别处），原预读计划立刻取消。

#### §4.2 预取下一集（跨文件）
- **触发：** 当前文件取流到一定比例（如 70%），且该文件所在目录有「下一集」（按文件名自然排序的下一个视频文件）。对下一集**首段（开头几个块）**调度 §3 预取进缓存。
- **目的：** 切下一集起播快——开头已在缓存零等待。
- **判定下一集：** 按文件名自然排序（`S01E01 → S01E02`），不靠影视数据库。
- **节流：** 预取优先级低于顺序预读；预取受 §2 缓存容量约束——水位高时预取让位于淘汰，宁少预取不超限。

#### §4.3 调度与节流（共用）
预加载器是唯一会主动触发 §3 的单元（主路径是播放器请求被动触发）。统一节流：
- **全局并发上限**：预加载触发的取流与主路径取流共享全局并发槽位池。主路径优先，槽位紧张时预加载让步。
- **每子源独立节流**：对同一子源的预加载请求数有上限，避免触发风控。
- **富余带宽才预加载**：决策器判定主路径已把上游跑满（吞吐接近画像带宽上限），预加载暂不做。
- **缓存命中跳过**：目标块已在缓存直接跳过。

**与 §1 闭环：** 预加载触发的取流同样产吞吐样本喂回 §1，补全冷门子源（平时不被主路径请求）的画像——预加载成了主动测速的副载体，帮 §1 解决「没被取流就没画像」的盲区。

**边界：** 只产「预加载计划」交调度器，不自己开连接、不自己写缓存。取消、让步、节流都在这层做决策，§3 只管执行拉取。

### §5 响应标记单元（Marker）

**做什么：** 把 §1 评估器的子源画像，标进「影视文件夹」名，让用户在播放器目录导航里看到源速度。

**「影视文件夹」判定：** 直接包含视频文件（按扩展名集合判定，如 `.mkv/.mp4/.ts/...`）的那个文件夹。列目录（`PROPFIND`）时，凡某文件夹下直接存在视频文件，即视为影视文件夹。不靠固定路径深度。

**唯一出口：虚拟改文件夹名。**
- `PROPFIND` 列目录响应里，对判定为影视文件夹、且其顶层子源「慢源标记 = true」的文件夹，在文件夹名后追加标记：`肖申克的救赎` → `肖申克的救赎·slow·1.5MBps`。快源下的影视文件夹不改名。
- 文件名不动 → 保住字幕匹配和类型识别。
- 标记字符用播放器友好集合，避开方括号、圆括号等通配/特殊字符（示例用中点 `·` 分隔，具体字符集留实现时按播放器实测微调，spec 不写死）。
- 代理维护「显示名 ↔ 真实路径」映射；取流时剥标记还原真实路径去上游取。

**已知风险，明确接受：** 影视文件夹名是刮削播放器（网易爆米花 Filmi 等）的匹配来源，加后缀可能破坏刮削或被刮削结果覆盖显示。这是用户已知晓并接受的取舍。文件名改名的方案已否决（刮削后显示数据库标题看不到文件名 + 破坏刮削）。

**不做：** 不改文件名；不真改上游文件夹名（绝不 `MOVE`）；不写响应头；不做状态页/查询接口/PROPFIND 自定义属性。

**依赖：** §1 评估器子源画像 + 取流实测吞吐。

**边界：** 只产标记，不路由、不取流、不缓存；映射只活在请求处理内存里，不持久化（重新列目录重算，画像变了标记自动更新）。

## 5. 数据流（端到端示例）

用户在播放器列目录 → 代理 `PROPFIND` 上游 → §5 对慢源下的影视文件夹虚拟改名返回 → 播放器显示 `肖申克的救赎·slow·1.5MBps`。

用户点播 → 播放器对代理发 `GET /肖申克的救赎·slow·1.5MBps/电影.mkv` Range 0- → 代理剥标记还原真实路径 → 路由层定子源 → §2 缓存查块 → 未命中 → §3 决策器首块探速 N=1 → 测到快追加 N=4 → 取流器 4 路并行拉分片乱序到顺序吐 → 边收边吐给播放器 → 每分片 `put` 缓存 → 吞吐样本回灌 §1。同时 §4 顺序预读指针后块、70% 时预取下一集首段。

## 6. 错误处理与降级

所有单元失败都透明降级，绝不主路径断流：

- **缓存层** 命中失败/磁盘满 → 降级不分片单路取流。
- **分片决策器** 未知子源 → N=1 起手探速，不冒进开多路。
- **取流器** 单路失败重试、重试耗尽补单路、最终 502（已成功块缓存保留，重试命中）；上游 429 多发 → §1 降级该子源友好度为风控 → 后续 N=1。
- **预加载器** 槽位紧张/带宽满/缓存水位高 → 让步或暂停，绝不与主路径抢。
- **标记单元** 画像缺失 → 该文件夹不改名（不标不是错）。

## 7. 测试策略

单元小、边界清晰，各自独立可测：

- **§1 评估器**：被动样本统计、主动探测多连接友好度判定、时效回退未知。
- **§2 缓存**：块命中/部分命中/未命中、LRU 淘汰、引用计数保护、高低水位、写前不超限、ETag 失效重建。
- **§3 取流器**：N 路乱序到顺序吐（核心）、首块探速动态扩路、重试与降级、429 反馈。
- **§4 预加载**：顺序预读触发与取消、下一集判定（自然排序）、全局/子源双节流让步。
- **§5 标记**：影视文件夹判定、虚拟改名与还原映射、只标慢源。

端到端：对真实 Alist 多驱动场景做集成测试（起播延迟、拖动、切集、慢源显示）。
