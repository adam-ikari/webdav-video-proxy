# 透明降级

代理的硬约束：**所有旁路失败都透明降级，绝不主路径断流**。播放器永远不应因为侧链故障而看到中断的流。

## 各场景的降级行为

| 场景 | 降级行为 |
|------|----------|
| 缓存命中失败 / 磁盘满 | 跳过缓存，直接走分片取流；写不进缓存就纯透传 |
| 上游不支持 Range（回 200 整文件） | 检测后降级为单连接整文件流，按块切片入缓存，仍吐出请求区间 |
| HEAD 请求失败 | 复用过期版本（如有）；否则 version 标 `unknown`，继续单路取流 |
| HEAD 返回的 size 与缓存不符 | 按 ETag 版本不匹配整文件失效重建 |
| 上游限流 429 / 503 | 该分片重试（上限 2 次，退避）；反馈 §1 把子源降级为风控，后续强制 N=1 |
| 某分片重试耗尽仍失败 | 该分片降级换单路重试；最终回 502 让播放器重试（已成功块缓存保留，重试时命中） |
| 淘汰器跟不上磁盘增长 | 超硬上限×1.5 后跳过新块缓存写入，绝不阻塞主路径 |
| 正在读取的块被淘汰器选中 | 引用计数保护，淘汰跳过该块 |
| 取流器某块迟迟未到 | orderedReader 等 30 秒超时后降级报错，不永久阻塞 |
| 预加载槽位紧张 / 带宽满 | 预加载让步或暂停，绝不与主路径抢 |
| 标记单元画像缺失 | 该文件夹不改名（不标不是错） |
| 未知子源（无画像） | 决策器保守 N=1 起手探速，不冒进开多路 |

## 为什么能保证不断流

- **主路径只走 §1 → §2 → §3 → 边吐**。§4（预加载）、§5（标记）在旁路，它们的失败不影响主路径。
- **缓存是优化不是必需**。命中失败 / 磁盘满 → 透明降级到不分片单路取流。
- **每个失败都有兜底**：重试、换策略、回错误码让播放器重试——但绝不静默 hang 住或吐半截流。
- **缓存命中路径不静默截断**：用单次 Get（而非 Has+Get 两段式）取块，缺失即回退 Fetch 回填，绝不吐少于 Content-Length 的字节。

## 播放器视角

播放器看到的是一个标准的、支持 Range 的 HTTP 资源。代理的所有加速、降级、缓存逻辑对播放器完全透明——它只感觉到「起播更快、拖动不卡、重复请求秒开」。
