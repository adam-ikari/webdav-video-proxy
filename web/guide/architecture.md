# 架构总览

WebDAV 视频代理是一个跑在 Docker 里的 HTTP 反向代理。它站在本地播放器和上游 WebDAV（如 Alist 聚合的多个网盘驱动）之间，对每个 Range 请求做测速、缓存、分片并行拉取，并预加载后续内容，从而消除播放卡顿。

## 子源模型

关键概念：**子源 = (上游端点, 顶级路径段)**。

一个聚合型 WebDAV（如 Alist）背后挂了多个真实驱动（阿里云盘、夸克、百度……），对外是「一个 WebDAV」，但不同顶层目录对应不同驱动，速度天差地别。代理按路径第一段目录把上游拆成多个**子源**：

```
https://alist.example.com
├── /阿里云盘/...   ← 子源 A（快）
├── /夸克网盘/...   ← 子源 B（中）
└── /百度网盘/...   ← 子源 C（慢）
```

测速、画像、分片决策、慢源标记全部落到**子源粒度**。每个文件只存在于唯一一个子源上——不存在「同一文件换更快的源取」这步。

## 请求主路径

每个播放器的 Range 请求依次过这几关：

```
播放器请求
   │
   ▼
路由层 ──§1 评估器(子源画像)──┐
   │                          │ (建议并发度 / 慢源标记)
   ▼                          │
缓存层(§2)──命中?──是────────►边收边吐──►播放器
   │ (否，块未命中)             ▲
   ▼                          │
§3 决策器(首块探速+动态扩路)──┐ │
   ▼                          │ │
§3 取流器(N路Range并行,乱序到顺序吐,回灌缓存)─┘
   │                          │
   └──(吞吐样本)──►§1 评估器──┘
                          ▲
§4 预加载器(顺序预读+预取下一集,节流让步)──调度──►§3取流器
§5 标记(列目录时按影视文件夹名改写)──读§1画像
```

1. **路由层** — 解析请求路径，确定该文件在哪个子源（映射固定，无择优）。
2. **缓存层 (§2)** — 块级命中检查。全命中直接吐、跳过分片。
3. **分片决策器 (§3)** — 首块探速 + 动态扩路，决定开几路并发。不支持多连接的源降级单连接。
4. **取流器 (§3)** — N 路 Range 并行拉分片，乱序到达、顺序吐出。
5. **边收边吐** — 透传给播放器，不整段缓冲再吐（避免起播延迟）。

## 旁路单元

以下单元在主路径之外，**任何失败都透明降级，绝不让播放器断流**：

- **§1 速率评估器** — 子源画像，喂给 §3 决策器（建议并发度）和 §5 标记（慢源标记）。
- **§4 预加载器** — 顺序预读 + 预取下一集，提前触发 §3 填 §2 缓存；与主路径共享并发槽位，主路径优先。
- **§5 标记单元** — 列目录（PROPFIND）时对慢源下的影视文件夹做虚拟改名。

## 设计取舍

- **不真改上游**：绝不调用 `MOVE`/`PROPPATCH`。所有「改名」都是代理对外展示的虚拟改名。
- **不写自定义响应头**：本地播放器（含网易爆米花等刮削播放器）不读 `X-*` 响应头，写了无用。
- **不做影视识别**：同名不同版本/码率的文件按路径当不同文件处理，不引入影视数据库。
- **不做跨源分片**：分片是「单源、单文件、多连接并行」，不是「不同源各拉一块拼」。
