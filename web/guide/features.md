# 核心功能

代理由五个边界清晰的单元组成。每个单元做什么、怎么用、依赖什么。

## §1 速率评估器（Rate Assessor）

**做什么**：对每个子源持续测速，维护一份画像（带宽档位、多连接友好度、建议并发度、是否慢源）。

**测速方式（主动 + 被动）**：

- **被动测速（主）**：每次真实取流顺带记一条吞吐样本到该子源的滚动样本集。零打扰。
- **主动探测（低频补盲区）**：子源首次出现、被动样本为 0 时主动探测一次给起步画像；以及多连接友好度判定（被动只按既有并发度跑，测不出「开更多路会不会更快」——靠主动「1 路 vs N 路」对照）。主动探测有最小间隔节流（默认 1 小时），不周期轰炸上游。

**慢源定义**：带宽档位低于阈值（默认 < 2 MB/s），或建议并发度被降为 1 且吞吐仍低。这个布尔值就是 §5 慢源标记的来源。

**画像时效**：最近 N 个样本 + 最大时效双约束（默认最近 20 个样本且不超过 6 小时）。样本不足或全过期 → 友好度回退「未知」→ 决策器保守单连接。

**出口**：① 喂 §3 决策器（建议并发度）；② 喂 §5 标记（慢源标记 + 带宽档位）。

## §2 缓存层（Cache）

**做什么**：把从上游取回的字节存到本地磁盘，下次同样 Range 命中就直接从本地吐、跳过分片取流。

**粒度**：块缓存，不整文件缓存。每个文件按固定块大小（默认 4MB）切，以块为单位存/缺失。视频常见「跳着播」「拖动进度条」，整文件缓存既浪费带宽又浪费磁盘；块缓存让被请求过的段热、没请求的段不拉。

**缓存大小管理**：

- **硬上限**（`CACHE_MAX_SIZE`，默认 50GB）+ **高水位**（90%）/ **低水位**（70%）。达高水位触发主动淘汰，低水位停止。
- **引用计数保护**：正在被取流/预加载使用的块（引用 > 0）淘汰跳过。
- **异步淘汰**：达高水位时不阻塞主路径，淘汰丢给后台 worker。
- **写前检查**：已到硬上限且淘汰跟不上则跳过这块写入（纯透传），宁可不缓存也不超限。
- **TTL 兜底**（`CACHE_TTL`，默认 7 天）：单块到期即使没达水位也淘汰。

**一致性**：缓存命中前低频向上游 HEAD 拿当前 ETag 比对 key 版本；不匹配整文件失效重建（见 §1 的低频校验，默认每文件 60 秒内不重复 HEAD）。

## §3 分片决策器 + 取流器（Segment Fetcher）

**决策器**做什么：产出一个拉取计划 `{起始字节, 结束字节, 并发度 N, 分块大小}`，喂给取流器。

**决策逻辑**：

1. 缓存全命中 → 不进决策器，缓存层直吐。
2. 多连接不友好/风控/未知 → `N=1`，单路取流。未知当不友好处理。
3. 多连接友好 → `N = 建议并发度`（默认上限 4）。
4. **首块探速 + 动态扩路**：即使画像说友好，也先用 1 路起手拉首块测实际吞吐；上游无报错则扩到 N，慢/报错保持 1。把「事后才知道速度」转化成「边取边调」。
5. **Range 支持检测**：首块拉取时若上游回 200 整文件（忽略 Range），立即降级为单连接整文件流。

**取流器**做什么：开 N 个 Range 子请求并行拉，每个拉一个分片。产出**按播放器 Range 顺序排列的 reader**，让边收边吐从头开始吐，不等所有分片到齐。

**关键难点 — 乱序到达、顺序吐出**：N 路并行，先返回的分片未必是播放器要的开头那段。取流器维护按字节偏移排序的待吐队列，吐出指针从 Range 起始字节往后推，遇到队列里有的块就吐、遇到还没到的块就等（带 30 秒超时，超时降级报错）。

**失败处理**：单路 Range 子请求失败（网络抖动、风控 429）→ 重试该分片（上限 2 次，退避）；429/503 反馈 §1 把该子源降级为风控，后续强制 N=1。

## §4 预加载器（Preloader）

**做什么**：在播放器还没请求前，提前把「马上要播的内容」拉进缓存。

**顺序预读（当前文件）**：播放器正请求某段，预加载器盯住当前吐出指针位置，提前把指针之后、同文件接下来的若干块调度给 §3 拉好塞缓存。拖动进度条/顺序播放时不卡。播放器跳走则取消旧预读计划。

**预取下一集（跨文件）**：当前文件取流到 70%，且所在目录有「下一集」（按文件名自然排序 `S01E01 → S01E02`），对下一集首段调度 §3 预取进缓存。切下一集起播快——开头已在缓存零等待。下一集用自己的版本号（HEAD 解析），不用当前文件的。

**节流**：预加载与主路径共享全局并发槽位池。主路径优先，槽位紧张时预加载让步。绝不出现「预加载占满带宽、播放器反而卡」。

## §5 标记单元（Marker）

**做什么**：把 §1 评估器的子源画像，标进「影视文件夹」名，让用户在播放器目录导航里看到源速度。

**「影视文件夹」判定**：直接包含视频文件（按扩展名集合判定，如 `.mkv/.mp4/.ts/...`）的那个文件夹。不靠固定路径深度。

**唯一出口 — 虚拟改文件夹名**：列目录（PROPFIND）时，对判定为影视文件夹、且其顶层子源标慢的，在文件夹名后追加标记：`肖申克的救赎` → `肖申克的救赎·slow·1.5MBps`。快源不改名。文件名不动（保住字幕匹配和类型识别）。标记用中点 `·` 分隔，避开方括号等通配字符。

代理维护「显示名 ↔ 真实路径」映射；取流时剥标记还原真实路径去上游取。

::: warning 已知取舍
影视文件夹名是刮削播放器（网易爆米花等）的匹配来源，加后缀可能破坏刮削或被刮削结果覆盖显示。这是为换取「在播放器目录里直接看到源速度」而接受的取舍。文件名改名的方案已否决（刮削后显示数据库标题，看不到文件名）。
:::
